[Issue 1023](https://github.com/w3c/webextensions/issues/1023): Proposal: filter argument for runtime.getManifest()

 * [rob] We add extension apis to fill gaps. Extensions can already use JS APIs to filter the returned object themselves. And from the browser engineering perspective, there is no benefit to implementing it either, because the synchronous shape of the API already requires the full data to be kept in memory across processes.
 * [carlos] Agreed. I don't think that filtering have any benefits. We better focus on alternatives for the method as listed in my comment on the issue.
 * [timothy] Agreed. It does not offer anything that you cannot do, nor cannot.
 * [simeon] The issue also states that `runtime.getManifest()` would be deprecated. I don't think that we were considering it.
 * [carlos] I think the idea was to limit the availability of this method to certain contexts.
 * [oliver] I think the intent was to have specific methods to solve target use cases.

[Issue 1024](https://github.com/w3c/webextensions/issues/1024): Scoped CustomElementRegistry fails with "Illegal constructor" in isolated content scripts

 * [simeon] I created the “topic: content scripts” and “topic: custom elements” labels to make it easier to keep track of issues like these in Github.
 * [simeon] Issue is that HTMLElement constructor works in the main world but not in content scripts.
 * [rob] Common thing is that it doesn't work across browsers. It certainly is not supported in Firefox. I recall an issue in Chrome, don't know if it's been fixed.
 * [oliver] Custom element registry doesn't work across worlds. Don't know that this is a bug so much as we don't have a clear path on how to make this work across worlds.
 * [rob] I created [PR 905](https://github.com/w3c/webextensions/pull/905) to specify extension contexts and isolated worlds, part of which is a suggestion to more formally establish and reference JS realms. I think that having grounding in JS / HTML specs is necessary if we want to change the behavior here.
 * [oliver] We don't have a resolution for the main custom element registry, but I think scoped registries may solve this.
 * [rob] General thought is we can't try to solve this if we are unfamiliar with the primitives. Right now we don't do anything special and there's no easy way to fix this without being very clear on what the behavior should be. Custom elements can have custom methods that can return values. What worlds should see what methods/getters? What should the behavior be when returning values? Automatic cloning raises more questions. Nontrival to solve.
 * [simeon] May need to talk to WHATWG.

[Issue 1025](https://github.com/w3c/webextensions/issues/1025): Request: Provide attach/detach event without debugger permission

 * [simeon] Request from Hilary from 1Password. She is not here, but there are other 1Password people here.
 * [benjamin] Agentic use cases. It would be nice if agents can reach into contexts and manipulate extensions. It would be nice if a tab could have a flag that tells whether the tab is being debugged.
 * [timothy] A tab might not start being debugged, but that wouldn't tell you that debugging state changes asynchronously after initial connection.
 * [carlos] tabs.onUpdated would fire.
 * [timothy] That may cover us then. I like the static property.
 * [oliver] Should clarify what's in scope with “debug” means. E.g. MCP server connections uses chrome devtools protocol over WebSocket, but in a comment Hilary mentioned wanting the extensionId, which is only available when debugger API is being used.
 * [benjamin] A bare metal version with isDebugging would be very useful. Can iterate from there.
 * [oliver] Other browsers don't support the debugger API, so this signal could be used for other situations like remote debugging.
 * [timothy] Right, we don't support debugger but we do support remote debugging. DevTools and webinspector are also susceptible to attacks.
 * [oliver] Breaking out devtools, debugging session, etc. might be useful.
 * [timothy] Also need to consider how you use this as an extension developer when attempting to debug your own extension.
 * [benjamin] Salient concern. In general we turn on and off hooks when in developer mode. Recently we became more proactive to warn people about weird contexts.
 * [timothy] Also need to consider web developers that are debugging their own websites and can't use the extension to log in.
 * [benjamin] We'd likely have our own heuristics to make decisions. Need a signal, from there we can iterate on how best to handle the situation.
 * [rob] Not sure the request here serves the use case. If we provide this flag, malicious extensions can work against debuggers (relatively minor). This would also not cover cases where another extension is being debugged (and use scripting API to modify the page for example). What is the specific use case you want to protect against? Phishing, agentic, something else? `externally_connectable` may be a better fit for agentic use cases.
 * [benjamin] If there's a debugger open (or some debugger API usage), we'd try to keep track of the context where someone is using it in order to have the option to limit access to our autofill.
 * [oliver] In Firefox and Safari, if there's a debugger attached it probably has the ability to attach to anything. If it can, then it could hook into the service worker to hide its work. We generally try to say that once a user has a debugger attached, we can't make many guarantees about limited access.
 * [benjamin] Agree that if you have malicious code running on your machine, that's bigger than what we can solve here. Not trying to stop the user, trying to provide the user with more information to help make a more informed decision.
 * [timothy] If that's the use case, a simple flag may be enough
 * [rob] A global flag may be better since there are multiple ways a debugger could modify tabs that are not directly related to “tab being debugged”. And e.g. in Firefox the whole browser can be debugged; I would not want to fire `tabs.onUpdated` for the flag change across potentially thousands of tabs.
 * [benjamin] Would this be flagged after any debug session starts?
 * [rob] Can discuss further.
 * [dave] Could this be a manifest option? Maybe something like “sensitive extension” to signal the extension has more concerns here.
 * [simeon] First thought when I read this was maybe “debugger.connection” as a permission. Sounds like we have tentative alignment on exploring a global flag.
 * [rob] Yes. Not yet supportive, but open to discussing. Hesitant about a tab-specific approach for the reasons stated already. Describe use cases so we can check whether the requested functionality covers it.
 * [benjamin] I will discuss with Hilary and bring it back here.

[Issue 1026](https://github.com/w3c/webextensions/issues/1026): extension.inIncognitoContext inconsistencies

 * [carlos] `extension.inIncognitoContext` doesn't behave consistently. Returns true or false depending on whether an extension context is in incognito / private browsing, but not true for all browsers. E.g. in incognito:spanning it is always false in extension popups.
 * [oliver] I believe it's intended or at least that we know that's how it currently works. Not sure if we'd be open to changing that.
 * [carlos] Would like to align to have more consistent behavior.
 * [rob] In the issue you stated that Chrome does not open extension tabs in incognito mode. That is not the case, I have written extensions that open extension tabs in incognito mode, when `incognito:split` is used and incognito access is granted.
 * [rob] In Firefox and Safari the behavior of private browsing is automatically managed, in Chrome incognito:split is required (which are not supported by Firefox/Safari). When incognito access is allowed, Chrome can open incognito content just fine. With incognito:spanning, Chrome loads everything in the same context, and `browser.extension.getViews()` can see each other.

[Issue 711](https://github.com/w3c/webextensions/issues/711): Proposal: Enable changing the permissions for only new user

 * Relevant proposals:
   * [PR 798](https://github.com/w3c/webextensions/issues/798): Proposal "Allow to change permissions behavior on extension updates
   * [PR 883](https://github.com/w3c/webextensions/pull/883): Proposal: initial_permissions and initial_host_permissions
 * [simeon] Different browsers handle permission changes differently. There is a desire to address this issue. The last major discussion we had was in the Berlin face-to-face meeting, where we roughly aligned on a new manifest key.
 * [simeon] I synced with Devlin (not PR 883), and he was concerned about introducing new fields to change behavior.
 * [carlos] Last conversation I had with Devlin was that he wasn't much in favor of either.
 * [carlos] backwards compatibility which can be addressed with 883, blurring concept/transition of making all host permissions optional by definition. Instead of “permissions” and “optional_permissions”, it seems more elegant to declare “permissions” and “initial_permissions”.
 * [oliver] Would be nice for there to be a comparison.
 * [simeon] Browser folks, strong opinions or preferences?
 * [rob] Currently a preference towards a manifest field stating how to handle it (PR 798). Firefox can do either, but there is no way for extensions to express their preference right now. PR 883 adds yet another set of permissions, which may reduce the comprehensibility of the extension API surface. I'm still open to arguments in favor of either proposal though.
 * [carlos] point of initial_permissions would be to eventually treat optional_permissions and permissions as the same thing and drop either moving forward. The concern with developer cognitive load is also a concern for 798.
 * [benjamin] I'd like the feature. I can see if I can write up a comparison.
